# UART 启动摘要被 printf 缓冲区截断

## 1. 问题标题

- `UART 启动摘要被单次 printf 缓冲区截断，提示符挤到状态行尾部`

## 2. 问题现象

- 上电后 `USART1` 能打印启动摘要
- 但摘要打印到 `DAC1_CH1` 那一行中间会被截断
- CLI 提示符 `> ` 直接接在半截状态行后面

示例现象：

```text
DAC1_CH1: state=running, mode=dc, amp_mv=500, offset>
```

## 3. 影响范围

- `USART1` 启动摘要显示
- CLI 首屏可读性
- `App/uart/uart.c`
- `App/uart/uart.h`

## 4. 复现条件

- 硬件平台：`STM32H743`
- 工程已启用 `DAC` 状态打印
- 上电后查看 `USART1` 启动摘要
- 当启动摘要总长度超过单次 `my_printf()` 的内部缓冲区时稳定复现

## 5. 根因分析

结论：

- 启动摘要最初是拼成一个长字符串，通过一次 `my_printf()` 发送
- `my_printf()` 内部使用固定长度缓冲区，超出后会被截断
- 后续 `CLI_ShowPrompt()` 继续输出 `> `，于是提示符看起来像贴在被截断的状态行后面

已确认事实：

- 问题不是 `CLI_ShowPrompt()` 少了换行
- 问题本质是单次格式化输出过长，被内部固定缓冲区截断

## 6. 解决方案

- 保留 `my_printf()` 作为普通状态输出接口
- 将启动摘要改为“按行分多次输出”
- 避免再把整段多行状态摘要拼成一次发送

这样做的原因：

- 修复目标更直接
- 对现有 CLI 输出逻辑影响最小
- 后续继续增加启动摘要字段时也更稳

## 7. 修改位置

- `zuolan_signal_STM32/App/uart/uart.c`
  - `UART_PrintBootStatus()` 改为逐行输出
- `zuolan_signal_STM32/App/uart/uart.h`
  - 保留较大的 `UART_PRINTF_BUF_SIZE`
- `docs/串口与CLI/UART接收调试与CLI运行说明.md`
  - 同步补充启动摘要输出注意事项

## 8. 验证方法

1. 重新编译并烧录
2. 打开 `USART1`
3. 上电观察启动摘要
4. 确认 `DAC1_CH1` 状态行完整输出
5. 确认 `> ` 提示符出现在新的一行

## 10. 注意事项 / 后续建议

- 后续如果再增加启动摘要字段，优先继续按行发送
- 不建议把大段多行日志重新改回单次 `my_printf()` 拼接输出
- 如果后续需要更长格式化输出，再单独重构 `my_printf()` 的内部缓冲策略
