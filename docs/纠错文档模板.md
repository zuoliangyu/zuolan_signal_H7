# 纠错文档模板

## 1. 问题标题

一句话写清问题名称。

示例：

- `STM32H743 串口 DMA 接收无回显问题`

## 2. 问题现象

描述用户可见现象，不要先写推测。

示例：

- 串口可以发送启动提示
- PC 发送 `hello` 后，板端没有回显
- 加断点后现象可能变化或暂时消失

## 3. 影响范围

说明问题影响哪些模块、外设、文件或功能。

示例：

- `USART1/USART2` 接收链路
- `DMA` 接收缓冲区
- 调试回显功能

## 4. 复现条件

描述问题出现所需的硬件、软件和操作步骤。

建议至少写：

- 硬件平台
- 工程配置
- 触发步骤
- 是否稳定复现

## 5. 根因分析

说明最终定位到的真实原因。

要求：

- 先写结论
- 再写分析过程
- 区分“已确认事实”和“曾经的怀疑项”

## 6. 解决方案

说明最终采用的修复办法。

建议写清：

- 改了什么
- 为什么这样改
- 有没有替代方案

## 7. 修改位置

列出具体修改到的文件和关键点。

示例：

- `App/uart/uart.c`
- `Core/Src/usart.c`
- `STM32H743XG_FLASH.ld`

## 8. 验证方法

说明修复后如何验证。

示例：

1. 重新编译并烧录
2. 上电观察启动日志
3. 向 `USART1` 发送 `hello`
4. 观察是否原样回显

## 9. 验证结果

记录验证结论。

允许两种状态：

- 已验证通过
- 尚未验证，仅完成代码修复

如果未验证，要写清原因。

## 10. 注意事项 / 后续建议

记录后续还可能踩坑的点。

示例：

- `STM32H7` 的 DMA 缓冲区不要放在 `DTCM`
- 如果后续开启 `D-Cache`，要补缓存一致性处理
- 如果重新生成 CubeMX 代码，要检查自定义段和 DMA 配置是否仍然保留
