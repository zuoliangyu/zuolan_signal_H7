# 05 常见误区与排查

## 1. 误区：多任务一上来就必须全开 Cache

不对。

RTOS 不是要求你必须立刻全开 `ICache + DCache + MPU 精细分区`。

真正需要你认真处理的是：

- 多任务 + DMA + 大缓冲区 + 算法

## 2. 误区：CubeMX 生成了 MPU_Config，我就不用再学 MPU

不对。

CubeMX 生成的是“基础模板”，不是你未来完整工程的最终策略。

一旦你开始做：

- ADC DMA
- 多缓冲
- 以太网
- SD 卡
- 音频/I2S/SAI

你几乎一定会继续改 Region。

## 3. 误区：只要把 DCache 打开，性能就一定更好而且没副作用

不对。

如果你没有处理一致性问题，DCache 打开后的典型表现是：

- DMA 数据偶尔错
- 算法输入偶发异常
- 发送内容旧一拍
- 加断点后反而“看起来正常”

## 4. 误区：Shareable 就等于自动一致

不对。

`Shareable` 是区域属性的一部分，但它不是“自动同步 CPU Cache 和 DMA”的魔法开关。

真正的一致性，还是要靠：

- non-cacheable 区
- 或 clean / invalidate

## 5. 误区：先把所有 Region 都配满，后面再说

不建议。

你现在更需要的是：

- 先建立一套稳定模板
- 再按需求增加 Region

否则你会陷入“选项很多，但不知道哪个在起作用”的状态。

## 6. 遇到诡异现象时怎么判断是不是 Cache 问题

下面这些现象很像 Cache 一致性问题：

- 不打断点错，打断点就对
- Debug 模式看起来正常，跑全速就异常
- 同一段代码换一个 buffer 大小后行为变化
- DMA 回调进来了，但数据内容不对
- 发送前明明改过 buffer，外设却输出旧数据

## 7. 一套简单排查法

### 第一步

先确认当前有没有开 `DCache`。

如果没开：

- 先别把锅甩给 DCache

### 第二步

确认 buffer 放在哪块内存。

### 第三步

确认是谁在写，谁在读：

- CPU 写 / DMA 读
- DMA 写 / CPU 读

### 第四步

如果开了 DCache：

- 接收链路优先怀疑 invalidate
- 发送链路优先怀疑 clean

### 第五步

如果暂时定位不清：

- 先把相关 buffer 放进 non-cacheable 区做对照实验

## 8. 最适合你当前阶段的策略

你现在不是要把 H7 的所有系统特性一次学完，而是要建立“不会把自己绕进去”的工程顺序。

所以当前最稳妥的判断标准是：

- 不确定时，先保守
- 先开 ICache
- DCache 等 DMA 策略清晰后再开
- MPU 从基础模板逐步演进，不要一次堆满
