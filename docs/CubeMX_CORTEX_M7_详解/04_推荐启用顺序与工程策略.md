# 04 推荐启用顺序与工程策略

这一篇不是讲“理论上都有什么可能”，而是直接讲你这个项目现在该怎么推进。

## 1. 先给一套可执行路线

### 阶段 A：工程起步

建议：

- `Speculation = Enabled`
- `ICache = Disabled 或准备开启`
- `DCache = Disabled`
- `MPU = 保留 CubeMX 基础模板`

目标：

- 先把时钟、GPIO、基础外设、下载调试链路稳定下来

### 阶段 B：基础性能提升

建议：

- 开启 `ICache`
- 仍然保持 `DCache = Disabled`

目标：

- 先拿到比较安全的性能收益
- 不引入 DMA 一致性复杂度

### 阶段 C：第一个 DMA 外设

建议：

- 先不开 `DCache`
- 先把 `ADC / UART / SPI` 的 DMA 数据链路跑通
- 先看清楚 buffer 放在哪、谁在读、谁在写

目标：

- 先把系统数据流走顺

### 阶段 D：进入真正的 H7 工程化

建议二选一：

1. 开 `DCache`，手动做 cache 维护
2. 开 `DCache`，并把 DMA buffer 配成 non-cacheable 区

目标：

- 既拿到性能
- 又保证一致性

## 2. 为什么我不建议你现在直接全开

因为你现在还在“建立模板工程”的阶段。

这个阶段最大的风险不是性能不够，而是：

- 你把一堆系统特性全打开
- 但还没建立稳定的排查模型
- 后面出了问题不知道是 GPIO、时钟、DMA、RTOS，还是 Cache

对于 H7，新手和中期阶段都更适合“分层打开”。

## 3. 只开 ICache 的价值

这是你现在最适合优先做的动作。

原因：

- 收益比较明确
- 不容易直接破坏 DMA 数据可见性
- 代码执行效率提升通常能直接感知

所以如果你问“现在第一步应该开哪个”：

- 首选 `ICache`

## 4. DCache 什么时候值得开

当你出现下面这些场景时，DCache 才开始显得有价值：

- FFT
- FIR/IIR
- 多块大数组运算
- 频繁处理采样缓冲
- RTOS 下多个任务共享大数据块

但一旦开了，你就必须明确回答这些问题：

- 哪些数组被 DMA 改写
- 哪些数组由 CPU 写、DMA 读
- 这些数组现在位于哪块内存
- 这些地址是否 cacheable
- 什么时候 clean / invalidate

如果回答不出来，就先别开。

## 5. 工程里最常见的两种成熟方案

### 方案 A：DCache 开启 + 手动维护

思路：

- 普通数据继续 cacheable
- DMA 收发前后做 cache maintenance

优点：

- 对 CPU 性能更友好

缺点：

- 容易漏写
- 需要严格遵守 buffer 对齐和地址范围要求

### 方案 B：DCache 开启 + DMA buffer 单独 non-cacheable

思路：

- 普通算法区用 cache
- DMA 共享区不用 cache

优点：

- 一致性更稳定
- 心智负担更低

缺点：

- 需要你会配链接脚本/段/MPU Region

## 6. 对你现在这个仓库的现实建议

我建议你当前这个仓库按下面节奏走：

### 现在

- 先不动 `DCache`
- 先继续把工程基础功能跑通
- 有空的话先理解当前 `MPU_Config()`

### 下一步

- 单独做一次“只开 ICache”的版本
- 观察系统是否稳定

### 再下一步

- 当你接入第一个 DMA 外设时
- 我们再专门做一版 `DCache + MPU` 策略文档和配置

## 7. 如果你后面上 RTOS，建议怎么分区思考

不要把所有数据都当成“一坨全局变量”。

至少分成四类：

1. 任务栈 / RTOS 控制块
2. 算法工作区
3. DMA 接收缓冲区
4. DMA 发送缓冲区

然后分别思考：

- 哪些适合 cacheable
- 哪些适合 non-cacheable
- 哪些需要单独段

## 8. 一套非常务实的最终建议

如果你只想记住一句话，就记这个：

- 现在先开 `ICache`
- `DCache` 等第一个 DMA 场景稳定后再开
- `MPU` 未来主要用来给 DMA buffer 单独划 non-cacheable 区
