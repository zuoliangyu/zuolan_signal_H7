# 串口配置与测试说明

本文档对应当前工程中这几部分：

- `Core/dma.*`：CubeMX 生成的 DMA 时钟和 DMA NVIC 初始化
- `Core/usart.*`：CubeMX 生成层，负责串口底层初始化、DMA 句柄和 NVIC
- `Core/stm32h7xx_it.*`：USART 中断和 DMA 中断入口
- `App/uart/uart.*`：应用层串口收发、环形缓冲区、测试任务

如果你想专门理解“为什么这个问题最后要改链接脚本，而不是只改 `.ioc`”，可继续看：

- `docs/STM32H7_DMA缓冲区与链接脚本说明.md`

## 1. 当前硬件配置

当前工程启用了两路异步串口：

- `USART1`
  - `TX = PA9`
  - `RX = PA10`
  - `115200 8N1`
  - `RX DMA = DMA1_Stream0`
  - `DMA Request = DMA_REQUEST_USART1_RX`
- `USART2`
  - `TX = PA2`
  - `RX = PA3`
  - `115200 8N1`
  - `RX DMA = DMA1_Stream1`
  - `DMA Request = DMA_REQUEST_USART2_RX`

当前 `.ioc` 也已经同步了这两路配置：

- `USART1_RX -> DMA1_Stream0`
- `USART2_RX -> DMA1_Stream1`
- `USART1 global interrupt`
- `USART2 global interrupt`
- `DMA1_Stream0_IRQn`
- `DMA1_Stream1_IRQn`

## 2. 当前软件结构

当前串口采用三层分工：

- `Core/dma.c/.h`
  - 打开 `DMA1` 时钟
  - 初始化 `DMA1_Stream0/1` 中断优先级
- `Core/usart.c/.h`
  - 初始化 `USART1/USART2`
  - 初始化 RX DMA
  - 关联 `huartx.hdmarx`
  - 打开 `USARTx_IRQn` 和 `DMA1_Streamx_IRQn`
- `Core/stm32h7xx_it.c/.h`
  - `DMA1_Stream0_IRQHandler()`
  - `DMA1_Stream1_IRQHandler()`
  - `USART1_IRQHandler()`
  - `USART2_IRQHandler()`
- `App/uart/uart.c/.h`
  - `my_printf()` 阻塞打印
  - `UART_Init()` 启动 `ReceiveToIdle + DMA`
  - `HAL_UARTEx_RxEventCallback()` 处理 `IDLE / HT / TC`
  - 软件环形缓冲区
  - `uart_proc()` 从环形缓冲区取数据并回显

这样分层后：

- `Core` 只管底层硬件和中断入口
- `App/uart` 只管串口应用逻辑
- 后面做协议解析时，不需要改 `Core` 的业务逻辑

## 3. 当前接收方案

这版已经不是简单阻塞测试，而是：

- `USART1/USART2` 都开启 `RX DMA + Circular`
- 使用 `HAL_UARTEx_ReceiveToIdle_DMA()`
- 使用 `HAL_UARTEx_RxEventCallback()` 接收事件
- 同时处理三类事件
  - `HAL_UART_RXEVENT_HT`：DMA 半缓冲
  - `HAL_UART_RXEVENT_TC`：DMA 满缓冲
  - `HAL_UART_RXEVENT_IDLE`：串口空闲

实际流程如下：

1. 串口字节先进入 DMA 接收缓冲区
2. DMA 到半缓冲时触发一次回调
3. DMA 到满缓冲时再触发一次回调
4. 串口总线空闲时也会触发一次回调
5. 回调里根据当前位置和上一次位置，计算新增数据段
6. 新增数据搬进软件环形缓冲区
7. `uart_proc()` 周期执行，从环形缓冲区取出数据并回显

## 4. 当前缓冲区设计

每个串口各自维护一套缓冲区：

- DMA 接收缓冲区：`256B`
- 软件环形缓冲区：`1024B`

其中 DMA 接收缓冲区已经显式放到链接脚本里的 `.dma_buffer` 段，并映射到 `RAM_D2`，原因是：

- 当前工程默认把普通 `.data/.bss` 放在 `DTCMRAM`
- `STM32H743` 的普通 DMA 不能访问 `DTCM`
- 如果把 UART DMA 缓冲区继续放在 `DTCM`，现象通常就是“串口发得出去，但 DMA 收不到，回显没反应”

设计目的：

- DMA 缓冲区负责“硬件连续收”
- 环形缓冲区负责“软件解耦”

这样即使主循环这几毫秒没空立刻处理，数据也不会马上丢在 DMA 这一层。

### 溢出策略

当前环形缓冲区满了以后：

- 新到的数据停止继续写入
- 置位 `overflow` 标志
- `uart_proc()` 会打印一次溢出提示

这版是调试优先方案，便于你先把链路跑通。后面如果你要做协议栈，可以再改成：

- 覆盖最老数据
- 统计丢包字节数
- 分协议通道做独立缓冲

## 5. 当前测试行为

`main.c` 里已经这样接入：

1. `MX_USART1_UART_Init()`
2. `MX_USART2_UART_Init()`
3. `UART_Init()`
4. 注册 `uart_proc()` 调度任务，周期 `1ms`

当前上电后行为：

- 不主动打印临时诊断信息
- 串口收到的数据会被原样回显到对应串口

例如你从 `USART1` 发：

```text
hello
```

就会在 `USART1` 收到回显：

```text
hello
```

`USART2` 同理。

说明：

- 之前联调阶段曾临时加入过启动状态和回调诊断输出
- 在确认回显链路正常后，这些临时诊断已经移除
- 当前版本保留的是正式回显行为，不再默认刷调试日志

## 6. 为什么这版先不打开 UART FIFO

虽然 `STM32H7` 的 UART 支持硬件 FIFO，但这版先保持：

- FIFO 阈值已配置
- FIFO 模式实际关闭

原因很简单：

- 你当前目标是先把 `IDLE + DMA Circular + RingBuffer` 跑稳定
- 这套链路本身已经足够做现阶段调试和基本收发
- FIFO 再叠上去，会让触发时机和调试观感更复杂

所以当前建议是：

1. 先把这版稳定跑通
2. 后面如果串口速率更高、CPU 更忙，再考虑打开 FIFO 做进一步优化

## 7. H7 上最重要的注意点：D-Cache

你现在工程里当前是：

- 开了 `I-Cache`
- 没开 `D-Cache`

所以这版 DMA 收串口没有缓存一致性问题。

但如果你后面开启 `D-Cache`，就必须注意：

- DMA 写内存时，CPU 看到的数据可能还是 cache 里的旧数据
- 这会导致串口回调已经触发，但软件读出来的数据不对

到那时必须至少做下面二选一：

1. 把 DMA 缓冲区放到 non-cacheable 区域
2. 在读 DMA 缓冲区前做 `D-Cache invalidate`

当前代码里 DMA 缓冲区做了 `32B` 对齐，就是为了给你后面处理 cache 留接口，但这并不等于已经解决 `D-Cache` 问题。

结论：

- 现在这版：没开 `D-Cache`，可以直接用
- 以后开 `D-Cache`：必须补缓存一致性方案
- 当前这版还额外避开了另一个 H7 坑：DMA 缓冲区没有放在 `DTCM`

## 8. my_printf 的定位

当前仍保留：

```c
int my_printf(UART_HandleTypeDef *huart, const char *format, ...);
```

特点：

- 内部用 `vsnprintf`
- 用 `HAL_UART_Transmit()` 阻塞发送
- 适合当前调试提示

注意：

- 它现在适合调试和少量状态输出
- 不适合高频日志
- 后面如果你日志量很大，建议改成 `TX DMA` 或独立日志缓冲

## 9. 当前适用范围

这版适合：

- 基础串口收发验证
- 调试打印
- 简单命令行
- 后续继续往协议解析扩展

这版还不适合：

- 超高频日志
- 大吞吐双向通信
- 已开启 `D-Cache` 但未处理一致性的场景
- 带复杂协议分包和粘包处理的正式协议层

## 10. 后续扩展建议

如果后面继续往下做，建议顺序是：

1. 先把这版回显联调稳定
2. 增加 `UART_Read()` / `UART_Available()` 这类接口
3. 在 `App/uart` 之上再封一层协议模块
4. 如果日志很多，再加 `TX DMA`
5. 如果启用 `D-Cache`，同步处理 DMA 缓冲区 cache 一致性
