# ADC 设计与实现说明

本文档用于规划当前工程中 `ADC` 部分的设计、配置、实现路径和调试重点。当前平台为 `STM32H743`。

## 1. 当前规划目标

当前先按“稳定采样链路优先”来规划，不直接一开始就叠加复杂触发和复杂算法。

建议目标顺序：

1. 跑通单通道基础采样
2. 跑通 `ADC + DMA` 连续搬运
3. 再决定是否上多通道扫描
4. 再决定是否引入定时器精确定频触发

这样做的原因：

- `STM32H743` 的 `ADC` 能力足够强，但联动越多，问题定位越慢
- 当前工程更需要一个能持续扩展的稳定基础版本

## 2. 推荐实现路线

### 第一阶段：单通道、软件触发、轮询或简单读取

适用场景：

- 先验证引脚、模拟输入、参考电压和量化结果是否正确

建议：

- 先只开一个 `ADC` 通道
- 不急着加 `DMA`
- 先读到稳定变化的原始值

目标：

- 先证明“采样值是对的”

### 第二阶段：单通道、连续采样、DMA 搬运

适用场景：

- 需要持续采样，并希望降低 CPU 参与度

建议：

- 采用 `ADC + DMA Circular`
- 应用层维护自己的采样缓存
- 如果后续会开 `D-Cache`，提前考虑缓冲区内存与缓存一致性

目标：

- 先证明“采样链路持续稳定”

### 第三阶段：多通道扫描

适用场景：

- 同时采多个模拟量

建议：

- 在确认单通道稳定后再开扫描
- 明确每个通道的采样顺序和缓存下标

目标：

- 先证明“通道顺序和数据映射没有错”

### 第四阶段：定时器触发

适用场景：

- 需要固定采样频率
- 需要与其他链路同步

建议：

- 在前面三阶段稳定后再引入
- 优先使用一个专门定时器做 `TRGO`

目标：

- 先证明“采样频率和触发链路准确”

## 3. `STM32H743` 上的默认建议

如果当前还没有明确业务约束，默认建议如下：

- 应用层目录：`App/adc`
- 基础版本：单通道 + DMA Circular
- 后续增强：定时器触发 + 多通道扫描

需要注意的芯片特性：

- `STM32H743` 的 `ADC` 性能较高，但模拟前端条件仍决定最终有效精度
- 如果后续用 `DMA`，缓冲区不能只看“能不能编译”，要确认实际内存可达性
- 如果未来启用 `D-Cache`，必须补一致性处理

## 4. 推荐的软件结构

建议后续按分层整理：

- `Core`
  - 保留 `CubeMX` 生成的 `adc.*`
- `App/adc`
  - 放 `ADC` 应用层封装
  - 放采样缓存、滤波、平均值、对外接口

推荐文件：

- `App/adc/adc_app.c`
- `App/adc/adc_app.h`

推荐职责：

- `ADC_Init` 类逻辑仍由 `Core` 初始化
- `App/adc` 负责：
  - 启动采样
  - 提供读取接口
  - 维护缓存
  - 做简单滤波或数据整理

## 5. 影响范围

后续开始实现 `ADC` 时，预计会影响：

- `.ioc`
- `Core/Inc/adc.h`
- `Core/Src/adc.c`
- `Core/Src/main.c`
- `App/adc/adc_app.c`
- `App/adc/adc_app.h`
- `CMakeLists.txt`
- `docs/ADC与DAC/ADC设计与实现说明.md`

如果加入 `DMA`，还可能影响：

- 链接脚本
- `Core/Src/dma.c`
- `Core/Src/stm32h7xx_it.c`

## 6. 风险

最主要风险如下：

- 模拟输入脚实际板级连接与预期不一致
- 采样时间过短，导致高阻信号读数不稳定
- 多通道扫描后通道顺序理解错误
- `DMA` 缓冲区放错内存
- 开启 `D-Cache` 后数据读到旧值

## 7. 验收标准

后续开始实现 `ADC` 时，建议至少满足：

- 能读取到稳定且随输入变化的原始值
- 如果使用 `DMA`，缓冲区内容持续更新
- 多通道时每个通道数据对应关系清晰
- 应用逻辑不堆进 `main.c`
- 文档同步更新

## 8. 调试建议

后续调试建议顺序：

1. 先验证引脚和电压输入是否正确
2. 再验证单次转换值是否合理
3. 再验证 `DMA` 是否连续搬运
4. 最后再验证滤波、换算和上层逻辑
