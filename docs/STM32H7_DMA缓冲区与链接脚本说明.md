# STM32H7 DMA 缓冲区与链接脚本说明

本文档专门解释一个容易混淆的问题：

- 为什么 `CubeMX` 里已经把 `USART + DMA` 配好了
- 但在 `STM32H7` 上，DMA 缓冲区问题最后还是经常要落到链接脚本

## 1. 先说结论

一句话结论：

- `CubeMX` 负责配置外设
- 链接脚本负责决定变量最终放到哪块内存

所以：

- `USART` 怎么开
- `DMA Stream` 怎么选
- `NVIC` 怎么勾

这些可以在 `CubeMX` 里完成。

但是：

- 某个 DMA 缓冲区到底放在 `DTCM`
- 还是放在 `AXI SRAM / RAM_D1 / RAM_D2 / RAM_D3`

这不是 `CubeMX` 外设参数本身能精确解决的，而是链接脚本和变量 section 的职责。

## 2. CubeMX 能解决什么

以串口为例，`CubeMX` 很适合处理这些内容：

- 开启 `USART1/USART2`
- 选择异步模式
- 配波特率、字长、停止位、校验位
- 配置 `RX DMA`
- 选择 `DMA1_Stream0 / DMA1_Stream1`
- 打开 `USART1_IRQn / USART2_IRQn`
- 打开 `DMA1_Stream0_IRQn / DMA1_Stream1_IRQn`
- 生成 `dma.c`、`usart.c`、`stm32h7xx_it.c`

也就是说：

- “外设如何工作”
- “中断是否打开”
- “DMA 通道怎么连”

这些都是 `CubeMX` 擅长的。

## 3. CubeMX 不擅长解决什么

`CubeMX` 不擅长解决的是这种问题：

- “这个特定的数组必须放到某一块指定 RAM”
- “只有 DMA 缓冲区要放 `RAM_D2`，但别的全局变量还想留在 `DTCM`”
- “我要新增一个自定义 section，比如 `.dma_buffer`”

原因很简单：

- `CubeMX` 主要描述的是外设初始化
- 变量最终落到哪块内存，是链接器把各个 section 摆到内存地图时决定的

这一步不是 `USART_InitTypeDef`、`DMA_InitTypeDef` 能表达的。

## 4. 为什么 STM32H7 特别容易遇到这个问题

因为 `STM32H7` 的内存不是一整块“通用 RAM”，而是分区的。

常见会看到这些区域：

- `ITCM`
- `DTCM`
- `RAM_D1`
- `RAM_D2`
- `RAM_D3`

它们的访问特性并不完全一样。

对 DMA 来说，最关键的一点是：

- 并不是所有 RAM 都能被所有 DMA 正常访问

尤其常见的坑就是：

- `DTCM` 对 CPU 来说很快
- 但很多普通 DMA 外设并不能访问 `DTCM`

于是就会出现一种非常典型的现象：

- 串口初始化正常
- DMA 初始化正常
- 中断也打开了
- 但是接收缓冲区里就是没有数据

代码看着都对，结果还是不工作。

## 5. 你当前工程为什么必须改链接脚本

你这个工程当前默认链接布局是：

- 普通 `.data/.bss` 放在 `DTCMRAM`

这意味着：

- 如果你定义一个普通全局数组
- 比如串口 DMA 接收缓冲区
- 它默认就会跟着 `.bss` 一起落到 `DTCM`

而你当前使用的是普通 `DMA1` 去做 `USART RX DMA`。

问题就在这里：

- `DMA1` 不能正常访问 `DTCM`

所以即使：

- `.ioc` 配对了
- `DMA Stream` 配对了
- `IRQ` 打开了
- `HAL_UARTEx_ReceiveToIdle_DMA()` 也调用了

还是会出现：

- 串口发得出去
- 但 DMA 收不到
- 最后不回显

这时问题就已经不是“外设没配好”，而是“缓冲区放错内存了”。

## 6. 为什么不能只靠 CubeMX MemoryMap 页面解决

这是很多人第一次接触 H7 时最容易误判的地方。

`CubeMX` 里的 `MEMORYMAP` 或相关页面，更多是：

- 展示芯片的内存结构
- 辅助某些系统配置
- 帮你理解内存区域

但它并不等于：

- 你可以在界面里逐个指定“这个 C 变量放哪块 RAM”

真正要精确控制单个变量位置，通常还是这两步：

1. 给变量加 section
2. 在链接脚本里把这个 section 映射到目标 RAM

## 7. 标准做法是什么

在 `STM32H7` 工程里，比较标准的做法是：

1. 在 `CubeMX` 里把外设、DMA、NVIC 配好
2. 在代码里给 DMA 缓冲区单独 section
3. 在链接脚本里把这个 section 放到 DMA 可访问 RAM

例如当前工程就是：

- 在 `.ioc` 里配了 `USART1/USART2 + RX DMA + IRQ`
- 在代码里把 DMA 缓冲区放进 `.dma_buffer`
- 在链接脚本里把 `.dma_buffer` 放到 `RAM_D2`

这三步缺一不可。

## 8. 为什么不直接把整个 `.bss` 都改到普通 RAM

当然也可以。

比如你可以把：

- `.data`
- `.bss`

都整体搬到 `RAM_D1` 或 `RAM_D2`。

这样做的优点：

- 简单
- 很多 DMA 相关问题会直接消失

缺点也很明显：

- 你失去了把高频 CPU 数据放在 `DTCM` 的机会
- 后面性能优化空间更小
- 对 H7 来说不够细致

所以更常见、更合理的做法是：

- 普通数据继续按工程设计放
- 只有 DMA 缓冲区单独分段

## 9. 当前工程采用的做法

当前工程采用的是“单独分段”的方式：

- 在代码里定义 `.dma_buffer`
- 在链接脚本里把 `.dma_buffer` 映射到 `RAM_D2`

这样做的好处：

- 串口 DMA 缓冲区可被 DMA 正常访问
- 不影响当前工程其它 `.data/.bss` 布局
- 后面继续加 SPI/I2C/ADC 等 DMA 外设时，也能复用这套思路

## 10. 实际开发时怎么判断该不该看链接脚本

遇到下面这些现象时，就不要只盯着 `.ioc` 了：

- 外设初始化看起来都对，但 DMA 就是不工作
- TX 正常，RX 异常
- 中断已开，但缓冲区一直没变化
- 加断点后现象变化很奇怪
- 芯片是 `STM32H7`
- 工程使用了 `DTCM`

这时优先检查三件事：

1. DMA 缓冲区在哪块 RAM
2. 这块 RAM 是否能被对应 DMA 访问
3. 如果开了 `D-Cache`，是否还缺 cache 一致性处理

## 11. 和当前串口问题的对应关系

这次串口问题的本质链条就是：

1. `.ioc` 已经把 `USART + DMA + IRQ` 配好了
2. 代码也已经启动了 `ReceiveToIdle_DMA`
3. 但 DMA 接收缓冲区默认落到 `DTCM`
4. `DMA1` 无法访问该区域
5. 所以收不到数据
6. 最终表现为“发送 `hello` 无回显”

因此这次改链接脚本不是“多绕了一步”，而是问题真正所在。

## 12. 最后结论

你可以把它记成一句工程经验：

- `CubeMX` 解决“DMA 这条通路有没有接通”
- 链接脚本解决“DMA 搬运的目标内存是不是它真的能到达”

对于 `STM32H7`，这两层必须分开看。
