# 第三阶段：内存、Cache、DMA、MPU

## 学习目标

这一阶段是 `STM32H743` 学习的分水岭。你如果想做稳定的信号处理系统，这一阶段不能只停留在“知道有这些名词”。

目标是建立以下能力：

- 能为不同类型的数据选合适的内存区
- 能处理 `DMA` 与 `D-Cache` 的一致性问题
- 能理解为什么需要 `MPU`
- 能为后续算法和数据链路做基本性能布局

## 为什么这一阶段对信号处理特别关键

信号处理工程通常有这些特征：

- 连续采样
- 大块数据缓冲
- 周期性算法处理
- 多次内存访问
- 对延迟和抖动敏感

如果内存布局和缓存策略不合理，表现通常是：

- 数据偶尔错乱
- FFT 结果异常
- 某些缓冲区正常，换个大小就出问题
- 打开优化或开 `D-Cache` 后行为变化

## 必须搞懂的内容

### 1. 哪些数据应该放在哪

建议先建立粗粒度原则：

- 高频访问、强实时的小块数据，优先考虑更快的区域
- 需要被 `DMA` 访问的缓冲区，优先考虑适合外设访问且一致性容易管理的区域
- 代码、系数表、查表数据、日志缓冲不要混为一谈

学习时先把数据分成四类：

- 算法中间变量
- 输入采样缓冲
- 输出结果缓冲
- 常量系数和查找表

### 2. D-Cache 一致性问题

你必须真正理解这个现象：

- `DMA` 把新数据写入内存
- CPU 读取时可能还看到缓存里的旧数据

或者反过来：

- CPU 先修改了发送缓冲
- 数据还没写回内存
- `DMA` 发出去的是旧内容

这类问题在高速采样和大块数据处理里非常常见。

### 3. Cache 维护操作

你需要知道什么时候考虑：

- `Clean`
- `Invalidate`
- 或直接把某块区域设成非缓存

这一阶段不要求你把所有细节背下来，但至少要能建立判断：

- 接收缓冲如何处理
- 发送缓冲如何处理
- 何时用手动维护
- 何时用 `MPU` 改区域属性

### 4. MPU 的用途

`MPU` 在这里不是“高级可选项”，而是系统整理工具。你要理解它的两个实际价值：

- 给某些 `DMA buffer` 配置更安全的访问属性
- 降低缓存一致性处理的复杂度

### 5. 链接脚本和段放置

你至少要看懂：

- 某个数组为什么被放到这个段
- 这个段最终映射到哪块内存
- 如何给 `DMA buffer` 单独建段

如果你完全不看链接脚本或 `map` 文件，后续会一直靠猜。

## 建议实验

### 实验 1：同一段采样缓冲放到不同内存区

内容：

- 使用同样的 `ADC + DMA` 工程
- 仅改变缓冲区放置位置
- 记录系统表现

观察点：

- 中断触发是否稳定
- 数据是否正确
- CPU 占用是否变化

### 实验 2：打开和关闭 D-Cache 对比

内容：

- 在相同采样场景下分别测试开关 `D-Cache`
- 比较数据一致性和性能差异

观察点：

- 错误数据是否出现
- 算法耗时是否变化

### 实验 3：半缓冲处理中的一致性实验

内容：

- 在半传输回调中读取刚采样的数据
- 观察是否出现旧数据或错位数据
- 尝试加入缓存维护操作

目标：

- 让你亲眼看到 `Cache` 不是抽象问题，而是实测问题

### 实验 4：为 DMA 缓冲单独规划区域

内容：

- 给 `DMA buffer` 单独分段
- 尝试用 `MPU` 或链接段隔离它

目标：

- 建立“哪些数据需要特殊管理”的工程意识

## 阶段内建议做法

推荐逐步推进，不要一次把所有优化都加上：

1. 先让采样工程在最简单布局下跑通
2. 观察缓冲区位置
3. 引入 `D-Cache`
4. 复现一致性问题
5. 通过缓存维护或区域属性解决
6. 固化成自己的模板工程

## 常见误区

- 误以为 `DMA` 只要配置通了，数据就一定对
- 看见偶发错误，首先怀疑算法，不先查缓存一致性
- 不区分“CPU 访问快”和“外设访问方便”
- 所有数组都随手定义成全局变量，不做段管理
- 完全不看 `map` 文件

## 阶段验收标准

完成本阶段后，你应该能做到：

- 解释 `DMA + D-Cache` 常见失效场景
- 为采样缓冲区设计基本的放置策略
- 能说出 `MPU` 在你的工程里准备解决什么问题
- 能看懂一个简单的链接段布局
- 能把“偶发错误”缩小到缓存、一致性、内存区这类系统问题
