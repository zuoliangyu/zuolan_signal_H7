# 第五阶段：实时系统与工程化

## 学习目标

这一阶段的目标是把前面的采样、缓存和算法整合成一个长期可维护的实时系统，而不是只在实验室条件下跑通一次。

你要建立的能力包括：

- 划清中断、采样、算法、输出各自的职责
- 控制延迟和抖动
- 避免因为日志、显示、通信而破坏采样稳定性
- 用工程方法定位问题

## 为什么信号处理项目必须补这一阶段

很多信号处理 demo 会停在这里：

- 数据采到了
- FFT 也跑了
- 串口也打出来了

但一加上更多任务，就开始出现：

- 丢采样
- 数据块错位
- 峰值偶发跳变
- 串口打印一多系统就抖

问题本质通常不是算法错，而是任务边界和调度设计不清。

## 必须搞懂的内容

### 1. 中断里做什么，不做什么

建议原则：

- 中断里只做最短路径工作
- 记录事件、切换缓冲、发通知
- 不在中断里做重算法、长时间打印、大量内存拷贝

### 2. 主循环或任务里做什么

算法处理、结果打包、通信输出应尽量放在主循环或任务中完成。你需要开始形成下面这种分层：

- 采样层
- 缓冲管理层
- 算法层
- 输出层

### 3. 是否引入 FreeRTOS

建议：

- 如果当前只有单条采样处理链路，可先不用 `RTOS`
- 当你开始同时做采样、算法、显示、通信时，再引入 `FreeRTOS`

引入后重点不是 API，而是任务边界：

- 采样触发由中断驱动
- 算法任务处理数据块
- 输出任务负责串口、屏幕或上位机协议

### 4. 性能测量

你必须学会量化，而不是凭感觉判断：

- 单块处理时间
- 最大中断延迟
- 周期抖动
- CPU 占用趋势
- 数据吞吐

### 5. 错误定位方法

建议建立一套固定排查顺序：

1. 先确认采样节拍是否稳定
2. 再确认缓冲切换是否正确
3. 再确认 `Cache / DMA` 一致性
4. 最后再怀疑算法本身

## 建议实验

### 实验 1：轻重任务分离

内容：

- 中断只置位标志或发通知
- 主循环处理半缓冲数据
- 比较与“中断里直接处理”的差异

目标：

- 亲自验证任务分层的意义

### 实验 2：加入串口或上位机输出压力

内容：

- 在算法处理后增加输出
- 逐步提高输出量

观察：

- 采样链路是否被影响
- 丢样或延迟是否出现

### 实验 3：引入 FreeRTOS

内容：

- 创建采样事件通知机制
- 单独建立算法任务
- 单独建立输出任务

目标：

- 建立基础实时系统结构

### 实验 4：建立性能统计

内容：

- 记录每个 block 的处理耗时
- 记录最大耗时和平均耗时
- 记录处理超期次数

目标：

- 为后续项目优化提供依据

## 工程建议

### 目录和模块建议

尽量按职责拆分：

- `bsp` 或硬件抽象层
- `drivers`
- `signal`
- `app`
- `debug`

### 配置管理建议

把这些参数集中管理：

- 采样率
- 缓冲区长度
- 通道数
- 滤波参数
- FFT 点数

### 调试建议

优先保留以下调试能力：

- 波形导出
- 运行时间统计
- 错误计数器
- 缓冲区状态观察

## 常见误区

- 中断里做太多事
- 把通信打印当作“无成本调试”
- 引入 `RTOS` 后只学 API，不做任务边界设计
- 没有任何性能统计，却直接开始优化

## 阶段验收标准

完成本阶段后，你应该能做到：

- 设计一条清晰的实时数据处理链
- 解释各模块的时序关系
- 用数据而不是感觉判断系统是否过载
- 在加入通信或显示后仍维持采样稳定
- 对一个异常现象给出结构化排查路径
