# 第一阶段：H743 与 F4 的核心差异

## 学习目标

这一阶段的目标不是写大量代码，而是建立系统认知。你需要知道为什么 `F4` 上跑得顺的工程，换到 `H743` 上可能因为内存区、`Cache` 或总线域设计不当而变得不稳定。

## 为什么这一阶段必须先学

做信号处理时，你很快就会碰到：

- 高速 `ADC + DMA`
- 大块缓冲区
- 实时算法处理
- 多外设并行搬运

这些问题在 `F4` 上往往还能“先跑起来再说”，但在 `H743` 上，如果系统层认知不够，后面会反复返工。

## 必须搞懂的内容

### 1. Cortex-M7 与 Cortex-M4 的差异

重点理解：

- `M7` 的执行性能更高
- `I-Cache` 与 `D-Cache` 会显著影响性能
- `FPU` 对浮点算法友好
- 指令和数据访问路径更复杂

你不需要在这一阶段深挖微架构细节，但必须接受一个事实：`M7` 引入了更高性能，也引入了更复杂的一致性问题。

### 2. H743 的内存结构

重点关注：

- `ITCM`
- `DTCM`
- `AXI SRAM`
- 不同 `SRAM` 区域

你要搞清楚两类问题：

- 哪些区域适合放代码
- 哪些区域适合放实时数据和 `DMA buffer`

在信号处理中，缓冲区怎么放，直接影响吞吐和稳定性。

### 3. 总线与域

重点理解：

- `D1 / D2 / D3` 域
- 外设和内存不一定在同一个性能路径上
- 某些外设访问某些内存区并不理想

学习时不要死记域的表格，重点是形成“访问路径不是统一的”这个意识。

### 4. 时钟树与性能配置

重点理解：

- 主频上去后，时钟配置比 `F4` 更容易出错
- `PLL` 来源和外设时钟来源要分清
- `Flash latency`、供电配置、系统时钟要匹配

如果这个基础不稳，后面所有采样率和实时性分析都不可靠。

### 5. Cache 的存在会改变你的开发习惯

这件事要尽早建立意识：

- CPU 看到的数据不一定是内存中最新的数据
- `DMA` 写完数据，CPU 不一定立刻读到新内容
- 某些工程必须做 `Cache Clean / Invalidate`

你现在先知道问题存在即可，详细处理放到第三阶段。

## 建议学习顺序

1. 阅读 `H743` 的数据手册，先看主频、存储、封装、外设概览
2. 阅读参考手册中系统架构、时钟、存储映射相关章节
3. 打开官方 `CubeH7` 工程，观察 `SystemClock_Config()`
4. 画出你自己的简化版内存示意图
5. 画出“采样数据从外设到算法”的路径图

## 建议实验

### 实验 1：最小系统和时钟确认

目标：

- 串口打印系统时钟
- 确认 `HAL_GetTick()`、定时器时基、串口波特率都正常

验收：

- 主频和外设时钟配置能说清来源
- 你能解释 `SystemClock_Config()` 的关键步骤

### 实验 2：内存区域观察

目标：

- 分别定义几个不同大小的数组
- 观察它们放在哪块内存
- 熟悉 `map` 文件或链接脚本里的区域分布

验收：

- 你知道大数组为什么不能随便放
- 你能初步区分“给算法用的数据”和“给 DMA 用的缓冲”

## 常见误区

- 误以为 `H743` 只是 `F4` 的主频升级版
- 看到例程能跑，就忽略系统时钟和内存分布
- 不区分代码区、普通数据区、`DMA` 缓冲区
- 没建立 `Cache` 概念就直接做高速采样

## 阶段验收标准

完成本阶段后，你应当能独立说明：

- `H743` 相比 `F4` 的 5 个关键差异
- `H743` 为什么更适合做复杂信号处理
- 为什么高速采样工程不能只看外设初始化
- 后面为什么必须单独学习 `Cache / DMA / MPU`
