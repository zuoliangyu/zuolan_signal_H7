# 实时信号处理流水线设计

工程已具备 ADC 周期采样、CMSIS-DSP 集成、FFT/FIR/IIR 三套基础能力。本文档说明把它们串成一条 **ADC → 滤波 → FFT** 流水线的设计方案、数据流、状态机和 CLI 接口。

## 1. 目标

提供一个可由 CLI 控制的实时管道，让用户：

- 选择是否启用滤波（`none` / FIR / IIR Biquad）
- 选择 FFT 长度（256 / 512 / 1024）
- 单次触发或持续运行，把结果（峰值 bin、频率、幅度）通过 USART1 输出
- 切换滤波器时自动清状态，避免污染

## 2. 数据流

```
TIM2 触发 ──► ADC1 (DMA 双半缓冲) ──► s_adc_dma_buffer[2*128]
                                          │
                       ADC ISR (HAL_ADC_ConvHalfCpltCallback / FullCpltCallback)
                                          │
                                          ▼
                          ADC_APP_QueueBlock 入块队列（已有功能）
                                          │
                                          ▼
                       ┌──────────────────────────────────────────────┐
                       │  scheduler 任务 dsp_pipeline_proc (1 ms)     │
                       ├──────────────────────────────────────────────┤
                       │  if (pipeline 未启用) return                 │
                       │  while ADC_APP_PopBlock 拿到一块 (uint16, 256)│
                       │      ├─ uint16 → float32                     │
                       │      │     可选去 DC：sample - 2048           │
                       │      ├─ 过滤波器（none/FIR/Biquad，按枚举）  │
                       │      ├─ 累积到 s_acc_buf[fft_len]            │
                       │      └─ 累积满：触发 FFT                     │
                       │              ├─ RFFT + cmplx_mag             │
                       │              ├─ FindPeak                     │
                       │              ├─ 输出：peak_bin/freq/mag      │
                       │              ├─ 累积器清空                   │
                       │              └─ 若 mode=ONESHOT，进入 IDLE   │
                       └──────────────────────────────────────────────┘
```

数据宽度变化：

```
ADC raw uint16 [0..65535]      ← 注意：当前 ADC 配置为 16-bit 分辨率（adc.c）
   │
   │  (粗去 DC) raw - 32768.0f （PL_ADC_MIDSCALE）
   ▼
float32 中心化样本（典型范围 [-32768, +32767]）
   │
   │  滤波（FIR/Biquad/旁路）
   ▼
float32 滤波后样本
   │
   │  累积满 fft_len 时
   ▼
(精确去 DC) 减帧均值 arm_mean_f32
   │
   ▼
float32 RFFT + 取模 → mag[fft_len/2]
```

去 DC 分两级：
1. **粗去 DC（按块）**：减 ADC 中点 32768，把数值拉回零附近，避免巨大 DC
   吃掉滤波器的浮点动态范围，也让 IIR 启动瞬态变小。
2. **精确去 DC（按帧）**：FFT 前用 `arm_mean_f32` 算累积器的均值再减，
   消除残余 DC，防止泄漏到 bin 1。

不做 `/65536` 归一化，FFT 幅度数量级跟随 raw，便于人眼对比。

## 3. 模块边界

`App/dsp/dsp_pipeline.{h,c}`，与现有模块的依赖关系：

```
dsp_pipeline.c
    │
    ├── 上游消费：ADC_APP_PopBlock, ADC_APP_GetSampleRateHz, ADC_APP_BLOCK_SAMPLES,
    │             ADC_APP_IsStarted
    │
    ├── 算法依赖：DSP_RFFT_Magnitude, DSP_FindPeak,
    │             DSP_FIR_F32_*, DSP_Biquad_F32_*,
    │             DSP_Window_Generate / CoherentGain / Name / FromName
    │
    ├── 系数来源：dsp_filters.h（不直接用 dsp_filters.c 的 state，自带 state buffer）
    │
    ├── 自检用：dac_app.h（仅 SelfTest 期间临时驱动 DAC，跑完还原现场）
    │
    └── 输出：my_printf 到指定 UART（DMA 异步）
```

`dsp_pipeline` 不持有 ADC 配置权（不改 ADC 采样率 / block 长度），只**消费** ADC 已经产出的块。
对 DAC 也只在 `DSP_Pipeline_SelfTest` 内临时操作（保存→改→还原），常规帧处理不动 DAC。

## 4. 关键状态

```c
typedef enum {
    PIPELINE_MODE_IDLE = 0,
    PIPELINE_MODE_ONESHOT,   // 跑一次后回 IDLE
    PIPELINE_MODE_STREAM,    // 持续跑直到 stream off
} pipeline_mode_t;

typedef enum {
    PIPELINE_FILTER_NONE = 0,
    PIPELINE_FILTER_FIR_MA5,
    PIPELINE_FILTER_FIR_TEMPLATE,
    PIPELINE_FILTER_BIQUAD_LP_2K,
    PIPELINE_FILTER_BIQUAD_TEMPLATE,
} pipeline_filter_t;

// 模块单例状态
static struct {
    pipeline_mode_t   mode;
    pipeline_filter_t filter;
    dsp_fft_size_t    fft_size;       // 256 / 512 / 1024
    uint8_t           dc_remove;      // 0 / 1
    uint16_t          acc_count;      // 已累积样本数 [0, fft_size]
    uint32_t          frame_seq;      // 第几帧
    uint16_t          output_rate;    // 每 N 帧打印一次（STREAM 下生效，ONESHOT 总是打）
    UART_HandleTypeDef *out_huart;

    dsp_fir_f32_t    fir;
    dsp_biquad_f32_t biq;

    // 加窗
    dsp_window_type_t window;
    uint32_t          window_len;     // 当前窗系数对应的长度
    float32_t         window_cgain;   // coherent gain = sum(w)/N

    // 自动测试用：silent=1 时 emit_frame 不打印单帧日志
    uint8_t           quiet;

    // 最近一帧结果（emit_frame 末尾填，给 fftdump / selftest 取）
    dsp_pipeline_result_t last_result;
} s_pl;
```

状态切换：

```
IDLE ──pipeline run────► ONESHOT
IDLE ──pipeline stream on──► STREAM
ONESHOT ──完成一帧──► IDLE
STREAM ──pipeline stream off──► IDLE

任何状态：
   pipeline filter <name>     → 切换 + 清 state + acc_count = 0
   pipeline fft <size>        → 切换 + acc_count = 0 + 重生成窗系数 + 重算 cgain
   pipeline window <name>     → 切换 + acc_count = 0 + 重生成窗系数 + 重算 cgain
                                （若所选窗的长度约束与当前 fft 不匹配，自动回退到 none）
   pipeline dc on|off         → 切换 + acc_count = 0
   pipeline rate <N>          → 仅改打印降频，不动累积器 / 算法状态
```

## 5. 缓冲区清单

```
s_acc_buf[1024]      累积输入（按最大 FFT 长度分配）   4 KB
s_fft_scratch[1024]  RFFT 复数中间结果                  4 KB
s_fft_mag[512]       幅度谱（fftdump / GetLastMag 也读这个） 2 KB
s_window_buf[1024]   当前窗系数（FFT/window 切换时重填）  4 KB
s_fir_state[N+127]   FIR state（N=最大 num_taps）       约 1 KB
s_biq_state[2*M]     Biquad state（M=最大 stages）      < 1 KB
─────────────────────────────────────────────
合计                                                    ~16 KB
```

均为 static 全局，放 .bss。RAM_D1（AXI SRAM）即可。

## 6. CLI 接口

```
pipeline status                 显示 mode / filter / window / cgain / fft / dc / rate / acc / seq / fs
pipeline filter <name>|?        切滤波 / 列出（none / fir_ma5 / fir_template /
                                 biquad_butter2_2k / biquad_template）
pipeline window <name>|?|test   切窗 / 列出 / 跑 4 窗对比测试
                                 可选名：none / hann / hamming / blackman / kaiser
                                 （kaiser 只在 fft=1024 时可用，否则自动回退到 none 并警告）
pipeline fft <256|512|1024>|?   设 FFT 长度（同时重生成窗系数）
pipeline dc on|off|?            去 DC 开关
pipeline rate <1..1000>|?       STREAM 输出降频：每 N 帧打印一次（ONESHOT 总是打）
pipeline run                    单次运行
pipeline stream on|off          持续运行
pipeline selftest               端到端硬件回环：自驱 DAC 1 kHz sine，跑 4 窗对比 + PASS/FAIL
                                 （前提：板上 PA4 ↔ PA0 跳线）

fftdump                         触发一次 oneshot，把整段 magnitude 谱以 CSV 输出
                                 （顶层命令，不在 pipeline 子树下，但走的是同一条链路；
                                  详见 docs/DSP/FFT加窗与系数模板.md §5）
```

每帧输出格式（一行，可被脚本解析）：

```
[PIPELINE] seq=12 filter=fir_ma5 window=hann fft=1024 fs=256000 Hz frame_mean=-0.3 \
           peak_bin=4 peak_freq=999.46 Hz mag=5075000.00 amp=10150000.00
```

字段说明：
- `mag`：RFFT 后的原始 magnitude，受窗 coherent gain 衰减
- `amp`：`mag / cgain`，对单频信号近似回到原始幅度
- `frame_mean`：当帧精去 DC 时减掉的均值；`dc=off` 时为 0

`pipeline status` 输出示例：

```
[PIPELINE] mode=stream filter=none window=hann cgain=0.500 fft=1024 dc=on rate=10 \
           acc=384/1024 seq=42 fs=256000 Hz
```

## 7. 异常与边界

| 情况 | 处理 |
|---|---|
| ADC 未启动 | `pipeline run/stream on/selftest` 返回错误码，不进入工作态 |
| 用户切 `fft <size>` 当前正在累积 | 丢弃当前累积，重生成窗系数，从下一帧开始攒 |
| 用户切滤波器在 STREAM 模式 | 清 filter state，不退出 STREAM，下一帧用新滤波 |
| 用户切 `window kaiser` 但 `fft != 1024` | 打印警告并自动回退到 `none`（窗模板长度固定 1024） |
| ADC 块队列被填爆（处理跟不上）| 由 ADC_APP 已有的 dropped_block 计数器统计，pipeline 不主动恢复；可用 `pipeline rate` 降低输出频率 |
| `RunOneshotBlocking` 超过 timeout | 强制回 IDLE，返回 `-2`，调用方（CLI / selftest）负责打印失败 |
| FFT 长度未就绪（DSP_Init 没跑）| `DSP_Init()` 在 main 启动时已调用，正常情况不会触发 |

## 8. 性能预算

@ 480 MHz CM7 + FPv5-D16 + ICache，DCache 关：

| 步骤 | 耗时（每 ADC 块 128 样本）|
|---|---|
| uint16→float32 + DC 减法 | ~3 µs |
| FIR 5-tap | ~5 µs |
| Biquad 1 段 | ~3 µs |
| 复制累积 | ~2 µs |

@ 累积满 1024 一次：

| 步骤 | 耗时 |
|---|---|
| 1024-pt RFFT_F32 | ~50–100 µs |
| arm_cmplx_mag_f32 (512 点) | ~10 µs |
| arm_max_f32 (512 点) | ~5 µs |
| my_printf 输出 | ~200 µs（取决于 UART 波特率）|

每帧 4 ms 预算下，纯计算 < 200 µs，余量大。

UART 输出**已改为 DMA 异步**（`my_printf` 推进环形缓冲，`HAL_UART_Transmit_DMA` 后台发），调用方零阻塞。每帧日志现在带 `window` / `frame_mean` / `amp` 等字段，约 160 字节/行 × 11.3 µs/字节 ≈ 1.8 ms @ 921600 baud，仍不够 250 帧/秒（需 4 ms/行）。这种情况下用 `pipeline rate <N>` 降频输出（每 N 帧打一次，例如 `rate 10` → 25 行/秒），或用 `uarttx` CLI 命令观察 `dropped` 计数判断是否要进一步压缩格式。

注意：`fftdump` 一次性输出 ~12 KB（fft=1024，每行 ~25 字节 × 512 行），CLI 实现里自带反压（每 8 行检查 ring 至少有 256 字节空闲），不依赖 `rate` 设置，详见 `docs/DSP/FFT加窗与系数模板.md §5`。

## 9. 与 cache 的关系

工程当前 ICache 开、DCache 关。该流水线**完全兼容当前配置**，不需要任何 cache/MPU 改动。

未来若启用 DCache：
- ADC DMA 缓冲（`.dma_buffer` → RAM_D2）必须配 MPU 设为非 cacheable，否则 CPU 拿到旧数据
- pipeline 的 `s_acc_buf` 等内部 buffer 在 RAM_D1，可继续 cacheable，无问题

详见 `docs/STM32H7_DMA缓冲区与链接脚本说明.md`。

## 10. 扩展指引：加新滤波器

1. 按 `docs/DSP/MATLAB滤波器设计与导出.md` 在 MATLAB 设计、生成系数
2. 在 `App/dsp/dsp_filters.h` / `.c` 加常量
3. 在 `dsp_pipeline.h` 的 `pipeline_filter_t` 枚举末尾加一项
4. 在 `dsp_pipeline.c::pipeline_apply_filter` 的 switch 加一条 case
5. 在 CLI handler 的名字解析里加一条 `strcmp` 分支
6. 重编译，CLI `pipeline filter ?` 自动列出新名字

## 11. 已知简化（先不做的事）

- 没有 overlap：每次 FFT 之间不复用样本，时间分辨率 = fft_len / fs
- 没有归一化：保留 raw 数量级，便于人工对比 ADC 实际幅度
- 没有自动增益：超出 ADC 范围的输入会饱和，由 ADC 硬件层面承担
- PSD 幅度补偿只用了 coherent gain（`sum(w)/N`），单频准；宽带 PSD 需 processing gain（`sum(w^2)/N`），按需扩展
- 没有同时输出多个频谱：实时帧只输出峰值 bin；全谱用 `fftdump` 一次性 CSV 导出（详见 §6 与 `docs/DSP/FFT加窗与系数模板.md §5`）

## 12. 测试路径

优先用内置自动测试，失败再降级到手动：

| 验证项 | 自动 | 方法 |
|---|---|---|
| 端到端硬件回环（DAC→ADC→FFT） | ✅ | `pipeline selftest`：自驱 DAC 1 kHz / 1V / 1.65V offset，跑 4 窗对比，验证 peak_bin ±2 / amp ±30% / 跨窗 ±15%（前提：PA4↔PA0 跳线） |
| 加窗一致性（cgain 补偿） | ✅ | `pipeline window test`：4 窗对比表，要求 amp 偏离 baseline ±15% 内 |
| Pipeline 单次通路 | ✅ | `pipeline run`，看 `[PIPELINE] seq=...` 行 peak_bin 是否合理 |
| 整谱形状 | ✅ | `fftdump` 拉 CSV，上位机绘图 |
| 滤波切换正确 | 半自动 | DAC 输 5 kHz、fft=1024、fs=256k → 切 `biquad_lp_2k` 应看到 mag 大幅下降 |
| 状态清零正确 | 半自动 | 切滤波/切窗后立即 `run`，输出无 NaN/Inf，前几个样本不会异常大 |
| 流式稳定 | 手动 | `stream on` 持续 1 分钟，配 `pipeline rate 10` 减压，看 `report` 里 dropped_block 是否上涨 |
| CLI 解析鲁棒性 | 手动 | 错参（`pipeline fft 999` / `pipeline filter xxx` / `pipeline rate 0`）应给提示，不崩 |
| DAC→ADC 首帧不稳 | — | `selftest` / `window test` 已内置 2 帧 warmup，手测时也要先丢 1~2 帧（详见 `docs/纠错/`） |
